### Correctness deep-dives (the parts I'd attack hardest)

1. **Coverage formula.** `coveragePct = tradesWithCache / totalOpenTrades * 100`. Both numerator and denominator use the same `status = 'open'` predicate and `COUNT(DISTINCT market_slug)`. INNER JOIN on slug returns at most one row per slug. If a single slug has two open trades (rare but legal — two `BUY` rows on different `outcome_token_id` for the same market), `COUNT DISTINCT` counts it once in both, so the ratio is consistent. ✓
2. **Alarm threshold.** `shouldAlarmCoverage` uses strict `<` against 80%. Exactly-at-threshold is healthy. Confirmed by test `treats exactly threshold as healthy (strict less-than)`. ✓
3. **History trim.** `saveCoverageHistory` trims with `slice(-COVERAGE_HISTORY_MAX)` BEFORE the JSON serialize. Bounded write. ✓
4. **JSON parse robustness.** `loadCoverageHistory` catches parse errors, rejects non-array, and per-element type-guards `ts` and `pct` as numbers. Cannot crash on garbage value in `poly_kv`. ✓
5. **Closed-only branch.** The `--closed-only` flag bypasses both the priority queue AND the coverage measurement. Correct — `--closed-only` is a maintenance mode that refreshes already-closed cache rows; we don't want it polluting the coverage history with partial measurements. ✓
6. **Concurrent cron risk.** Coverage history write uses `INSERT ... ON CONFLICT(key) DO UPDATE`, a single-statement SQLite transaction. Two simultaneous fetcher runs would last-writer-wins on history; harmless given the cron is single-instance. ✓
7. **Migration safety.** No new migration. `poly_kv` is created on-demand by the script via `CREATE TABLE IF NOT EXISTS`. Existing repo pattern (matches `strategy-engine.ts:274`). ✓
